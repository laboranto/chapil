# 백엔드 정리 및 백업 시스템 재설계 — 설계 문서

**날짜:** 2026-07-16
**프로젝트:** chapil (차량 관리 앱)
**범위:** FastAPI 백엔드 다이어트 + 복구코드 기반 백업 시스템 도입

---

## 배경

프론트엔드(`frontend/src/api.js`, `db.js`)는 이미 대시보드·주유/정비/기타 기록·설정·이미지·가져오기/내보내기까지 전부 `sqlite-wasm` 기반 클라이언트 DB로만 동작한다. PWA와 Capacitor 네이티브 앱 모두 동일한 `getDB()` 경로를 타며, `fetch('/api/...')` 로 데이터를 주고받는 코드는 어디에도 없다(`/api/demo-seed` 제외).

반면 `app/main.py`(1157줄)는 fuel/maintenance/other CRUD, settings, import/export, 이미지 업로드 라우트를 여전히 갖고 있지만 지금 프론트엔드 어디서도 호출되지 않는다. README가 설명하는 "앱 실행마다 자동 백업"은 이 죽은 서버 DB를 백업하는 것이라 사실상 빈 껍데기를 백업하고 있었다. JS 쪽에 준비된 `autoBackupToFile()`(Android MediaStore 대상)도 실제로는 어디서도 호출되지 않는 죽은 코드였다.

이란토 본인의 실사용 환경은 Safari "홈 화면에 추가" PWA다. iOS Safari는 File System Access API를 지원하지 않아 브라우저에서 폴더에 조용히 자동으로 쓰는 방법이 없고, PWA는 OS 레벨 백업/복원 대상도 아니다. 즉 지금 실질적인 데이터 유실 위험은 네이티브 앱보다 이 PWA 사용 패턴에 있다.

---

## 목표

1. `app/main.py`에서 프론트엔드가 더 이상 호출하지 않는 코드를 제거하고, 백엔드가 실제로 하는 일(정적 서빙, 데모 시딩, 복구코드 저장)만 남긴다. (COOP/COEP 헤더는 실제로 필요한지 섹션 A에서 재검토)
2. PWA를 포함한 모든 플랫폼에서 동작하는 자동 백업/복구 수단을 마련한다. 방식은 냥코대전쟁류의 "복구코드" 모델 — 로그인 없이, 코드 하나로 자체 호스팅 서버에 암호화된 백업을 올리고 나중에 같은 코드로 복원한다.
3. `docs/privacy.md`를 실제 동작과 일치하도록 갱신한다. 현재 "외부 서버로 전송되지 않으며, 개발자를 포함한 누구도 열람할 수 없습니다"라는 문구는 복구코드 백업 도입 후 사실과 어긋나게 된다.

## 아키텍처 결정

- **백엔드는 상태 없는 정적 서버로 축소되지 않는다.** 복구코드 백업을 저장할 최소한의 영속 저장소(`recovery_backups` 테이블 하나)가 새로 필요하다. 기존 fuel/maintenance/other 스키마를 대체하는 것이지, "상태 없음"으로 가는 것이 아니다.
- **복구코드 백업 메커니즘(암호화/서버 API) 자체는 PWA 전용이 아니라 전 플랫폼 공통이다.** 순수 JS(`fetch` + Web Crypto API)로 구현되므로 새로운 네이티브 플러그인이나 플랫폼별 백업 코드(iOS 폴더 피커, Android SAF 개선 등)를 만들 필요는 없다. 단, **PWA(Safari)는 백엔드와 동일 오리진이라 상대경로 `fetch('/api/recovery/...')`가 그대로 동작하지만, Capacitor 네이티브 빌드는 웹뷰가 로컬 번들 오리진(`capacitor://localhost` 등)에서 로드되므로 상대경로가 Tailscale 백엔드에 닿지 않는다.** (`main.jsx`의 기존 `/api/demo-seed` 호출이 네이티브 빌드에서 이미 같은 이유로 조용히 실패 중 — `.catch(() => null)`로 무시되고 있었다.) 네이티브 빌드에서 실제로 이 기능을 쓰려면 절대 URL(백엔드 호스트) 설정이 별도로 필요하며, 이란토 본인은 PWA만 사용하므로 이 설정은 이번 세션 스코프 밖으로 미룬다.
- **기존 로컬 내보내기(`exportToFile`)와 Android `FilePlugin.autoBackup`은 삭제하지 않고 보조 수단으로 남긴다.** 자체 호스팅 서버가 장기간 닿지 않을 때의 대비책이며, iOS Safari에서도 다운로드 시트를 통해 iCloud Drive/Nextcloud 등 Files 연동 폴더에 수동으로 저장할 수 있다.
- **암호화는 대칭키로 처리하고, 조회 키와 암호화 키를 코드에서 서로 다른 방식으로 파생시킨다.** 서버 DB가 통째로 유출되어도 코드 자체(네트워크 캡처 등)가 함께 새지 않는 한 복호화가 불가능하도록 한다.
- **클라우드 커넥터(WebDAV/iCloud/Google Drive 직접 연동), 코드 이중화(확인코드), 쓰기 요청 rate limiting은 이번 스코프 밖이다.** 지금은 Tailscale VPN 뒤 단일 사용자 환경이라는 전제로 최소 구현만 한다. 공개 배포를 고려하는 시점에 다시 다룬다.

## 섹션 A: 백엔드 정리

### 제거 대상
- `app/database.py`의 기존 스키마(fuel/maintenance/other/settings 테이블), `sqlite3` 기반 CRUD 전부
- `main.py`의 fuel/maintenance/other CRUD 라우트, settings 라우트, import/export 라우트(`/api/import/*`, `/api/export`), 이미지 업로드 라우트(`/api/settings/image*`)
- 관련 Pydantic 모델(`FuelBody`, `MaintenanceBody`, `OtherBody`, `SettingsBody`, `Import*` 등), `CarType`/`CarFuel` Enum
- Redis 연결 및 fuel_list 캐싱 로직 (`redis_client`, `redis` import)
- **`backup_if_needed()`, `build_export_data()`, `seed_demo()`(Python 서버 시작 함수) — `lifespan()`이 매 기동마다 무조건 호출하는 함수들.** 전부 옛 fuel/maintenance/other/settings 스키마를 조회하므로, 이 함수들을 `lifespan()` 호출부에서 함께 제거하지 않으면 스키마 교체 후 서버가 `no such table: settings` 에러로 기동조차 안 된다. `seed_demo()`는 서버 DB에 데모 데이터를 미리 채우는 용도였는데, 이제 데모 시딩은 프론트가 `/api/demo-seed` JSON을 받아 클라이언트 DB에 직접 채우므로(`main.jsx`) 서버 쪽 시딩 자체가 불필요해졌다.
- `Dockerfile`의 `pip install` 목록(`requirements.txt` 파일은 없고 Dockerfile에 직접 나열됨)에서 `redis`, `openpyxl`, `pillow` 제거. `python-multipart`는 라우트 제거로 더 이상 필요 없으면 함께 제거

### 유지 및 신규
- `StaticFiles` 마운트(React 빌드 서빙) — 그대로 유지
- `CoopCoepMiddleware` — **재검토 필요(확인 필요).** main.py 주석은 "OPFS의 SharedArrayBuffer 때문에 필수"라고 되어 있지만, 실제 `frontend/src/db.js`는 OPFS를 전혀 쓰지 않고 항상 `sqlite3.oo1.JsStorageDb('local')`(localStorage 기반 kvvfs)을 사용한다 — OPFS 시도조차 하지 않는다. 즉 이 미들웨어의 존재 근거가 코드와 맞지 않는 낡은 주석일 가능성이 높다. 구현 시 실제로 헤더를 제거해도 kvvfs가 정상 동작하는지 테스트해보고, 정말 불필요하면 함께 제거하고 주석도 정정한다. 불확실하면 안전하게 유지.
- `/api/demo-seed` — 그대로 유지(파일 직접 읽기, DB 불필요)
- `app/database.py`를 신규 스키마로 교체:
  ```sql
  CREATE TABLE IF NOT EXISTS recovery_backups (
      lookup_key       TEXT PRIMARY KEY,   -- SHA-256(코드) 의 hex
      ciphertext       TEXT NOT NULL,      -- base64(IV + AES-GCM 암호문)
      retention_months INTEGER,            -- NULL = 무기한. 사용자가 최초 설정 시 선택
      updated_at       TEXT DEFAULT (datetime('now', 'localtime'))
  );
  ```
- 신규 라우트 3개:
  - `PUT /api/recovery/{lookup_key}` — body: `{ ciphertext, retention_months }`. upsert(`retention_months`는 매 푸시마다 함께 전달되어 사용자가 나중에 설정을 바꾸면 다음 푸시에 반영됨).
  - `GET /api/recovery/{lookup_key}` — 존재하면 `{ ciphertext, updated_at }`, 없으면 404.
  - `DELETE /api/recovery/{lookup_key}` — 사용자 본인 요청에 의한 수동 삭제. 코드를 아는 것 자체가 이미 읽기/쓰기 권한을 의미하므로 별도 인증 없이 조회키만으로 삭제 허용.
- 매 `PUT` 요청마다(별도 스케줄러 없이 기존 트래픽에 편승) `retention_months IS NOT NULL AND updated_at < datetime('now', '-' || retention_months || ' months')`인 행을 함께 정리한다. 즉 무기한을 선택한 사용자의 데이터는 절대 자동 삭제되지 않고, 기간을 선택한 사용자의 데이터만 대상이 된다.

`lifespan()`은 `init_db()` 호출 하나만 남긴다(신규 스키마 생성). 결과적으로 `main.py`는 약 60~100줄 수준(FastAPI 앱 + 미들웨어 + 정적 서빙 + 라우트 3개)으로 줄어든다. Docker/docker-compose 배포 구조 자체는 변경하지 않는다.

## 섹션 B: 복구코드 백업 시스템

### 데이터 흐름
```
[클라이언트] exportData() → JSON
           → AES-GCM 암호화 (키: HKDF(코드))
           → PUT /api/recovery/{SHA-256(코드)}  { ciphertext }

[복원 시]  GET /api/recovery/{SHA-256(코드)}
           → AES-GCM 복호화 (키: HKDF(코드))
           → 기존 importConfirm() 파이프라인 재사용
```

### 코드 및 키 파생
- 코드는 클라이언트에서 `crypto.getRandomValues()`로 생성한 32바이트(256비트) 무작위 값을 hex로 표현(64자). 기억하지 않고 복사/캡처해서 보관하는 용도이므로 넉넉한 엔트로피를 사용한다.
- 이후 모든 파생 연산의 입력은 **이 64자 hex 문자열을 UTF-8로 인코딩한 바이트열**로 고정한다(원본 32바이트가 아니라 hex 문자열 자체). 복원 시 사용자가 입력창에 붙여넣는 것도 동일한 hex 문자열이므로 별도 디코딩 단계 없이 그대로 쓸 수 있다.
- `lookup_key = SHA-256(코드 문자열)` — 서버에 조회 키로 전달. 서버는 이 값 외에 코드 원본을 알 수 없다.
- `encryptionKey = HKDF(코드 문자열, salt: 빈 바이트열 고정, info: "chapil-backup-v1")` → AES-GCM 256비트 키로 추출(`crypto.subtle.deriveKey`). salt를 고정값으로 둬야 복원 시 동일한 키를 재현할 수 있다(무작위 salt를 쓰면 별도로 저장/전달해야 하므로 여기선 코드 자체가 유일한 비밀이라는 전제하에 고정 salt로 충분하다). `lookup_key`와 다른 파생 경로(`info` 값이 다름)이므로, DB가 유출돼도 코드 자체 없이는 `lookup_key`로부터 `encryptionKey`를 역산할 수 없다.
- 암호화마다 랜덤 IV(12바이트)를 새로 생성하고 `base64(IV + 암호문)` 형태로 저장/전송한다.
- 전부 브라우저 표준 `crypto.subtle` API로 구현하며 외부 라이브러리는 추가하지 않는다.

### 클라이언트 동작
- 최초 실행 시 별도 토글 없이 자동으로 코드를 생성해 로컬에 저장한다(로그인 없는 모델과 일관되게, 켜고 끄는 상태를 만들지 않는다).
- 코드 생성 직후 1회성 고지 모달을 띄운다: 생성된 코드 표시 + 복사 버튼 + "서버에는 이 코드로 암호화된 데이터만 저장되며, 관리자를 포함해 코드 없이는 아무도 열람할 수 없다. **이 코드는 지금 기기 안에만 저장되어 있다 — 브라우저 저장소가 초기화되거나 앱을 지우면 코드 자체도 함께 사라지므로, 지금 바로 복사해서 별도로(메모 앱, 다른 기기 등) 보관해야만 나중에 복원이 가능하다**"는 안내 + "확인" 버튼으로 닫음. 이 고지는 기능을 껐다 켰다 하는 토글이 아니라 최초 1회 인지시키는 용도이며, 이후에는 설정 화면에서 코드를 다시 확인만 할 수 있다. (복구코드 백업이 막아주는 건 "기기는 멀쩡한데 로컬 DB만 손상/삭제된 경우"이며, "저장소 초기화·기기 분실"까지 복구하려면 코드를 미리 외부에 저장해뒀어야 한다 — 이 모델의 본질적인 전제다.)
- 같은 모달에서 **보존 기간을 사용자가 직접 선택**하게 한다: "무기한 보관(기본값)" / "1년 미접속 시 자동 삭제" / "6개월 미접속 시 자동 삭제" 중 택1. 기본값을 무기한으로 두는 이유는, 이 기능의 본질이 재해복구(오래 안 켜다가 기기를 잃어버린 뒤 돌아오는 시나리오 포함)인데 미접속 기준 자동 삭제를 기본으로 걸면 정확히 그 시나리오에서 안전망이 사라지기 때문 — 데이터를 최소한만 남기고 싶은 사용자는 직접 더 짧은 기간을 선택할 수 있게 열어둔다. 설정 화면에서 언제든 변경 가능하며, 변경 사항은 다음 자동 푸시 때 서버에 반영된다.
- 앱 시작 시 `maybeAutoBackup()`을 호출: 마지막 푸시 시각이 24시간 이상 지났으면 백업을 푸시한다(기존 Python `backup_if_needed()`의 24시간 스로틀 로직을 그대로 계승). 서버가 Tailscale로 닿지 않으면 조용히 실패하고 다음 기회에 재시도한다 — 오프라인 상태에서도 로컬 CRUD 자체는 100% 그대로 동작하며, 이 백업 푸시만 기회주의적으로 이루어진다.
- 설정 화면에 "코드로 복구" 입력창을 추가: 코드를 입력하면 서버에서 암호문을 받아 복호화하고, 기존 가져오기(import) 화면/로직을 그대로 재사용해 미리보기 후 반영한다. **미해결 open question: 기기에 이미 로컬 데이터가 있는 상태에서 복원할 때 병합/중복/전체 대체 중 어떤 정책을 취할지는 여기서 정하지 않았다 — 구현 계획에서 확정.**

### 보조 수단 (변경 없음)
- 수동 "내보내기"(`exportToFile`)는 그대로 유지 — 서버가 장기간 닿지 않을 때의 대비책이며, iOS Safari에서도 다운로드 시트를 통해 iCloud Drive 등에 수동 저장 가능하다.
- Android `FilePlugin.autoBackup()`은 존재하는 채로 두되, 이번 세션에서 새로 연결하지 않는다(복구코드가 이미 Android에서도 동일하게 동작하므로 우선순위 낮음).

### 보존 정책
- 법이 강제하지 않더라도(개인정보보호법상 "1년 미접속 시 자동 파기·분리보관" 의무는 2023.9.15. 시행 개정으로 삭제됨), 사용자에게 데이터 보존 기간을 직접 선택하게 한다 — 이건 법적 최소 요건이 아니라 사용자 자기결정권을 존중하는 방향의 자체 정책 결정이다.
- 선택지는 무기한(기본값) / 1년 미접속 시 자동 삭제 / 6개월 미접속 시 자동 삭제 세 가지. 서버는 매 `PUT` 요청에 편승해 조건에 맞는 오래된 행을 정리한다(위 스키마/라우트 참조). 별도 스케줄러나 cron은 두지 않는다.
- 그와 별개로 **수동 삭제**(`DELETE /api/recovery/{lookup_key}`)를 항상 제공한다 — 보존 기간과 무관하게 사용자가 언제든 즉시 지울 수 있다.
- `recovery_backups` 테이블은 조회키(코드의 SHA-256 해시), 암호문, 보존 기간 선택값, 갱신 시각 외의 컬럼을 두지 않는다 — 이메일·로그인 계정 등 사용자 식별 정보를 애플리케이션 레벨에서 추가로 수집하지 않는다. (uvicorn/리버스 프록시의 인프라 레벨 접속 로그는 이 정책과 별개이며 이번 스코프에서 다루지 않는다.)

## 섹션 C: 개인정보 처리방침 갱신

`docs/privacy.md`의 "이 데이터는 외부 서버로 전송되지 않으며, 개발자를 포함한 누구도 열람할 수 없습니다"는 두 절 중 **앞 절("전송되지 않으며")만** 복구코드 백업 도입 후 사실과 어긋나게 된다. 뒤 절("개발자를 포함한 누구도 열람할 수 없습니다")은 암호화 설계 덕분에 여전히 참이므로 그대로 유지하고, 앞 절만 고친다. 다음 내용을 반영해 갱신한다.

- **수집 정보**: 기본 저장은 여전히 기기 내부. 복구코드 백업은 **별도 설정이나 동의 절차 없이 최초 실행 시 자동으로 활성화되며 끌 수 있는 기능이 아니라는 점을 명시**(조건부 "사용 시"가 아니라 상시 동작임을 분명히 서술). 24시간마다 한 번씩 암호화된 데이터가 개발자 운영 서버로 전송됨을 설명. 암호화 키는 코드로부터만 계산되고 서버에는 저장되지 않으므로, 개발자를 포함해 코드를 모르는 누구도 열람 불가능함을 설명.
- **수집 목적**: 로컬 DB 손상/삭제 등으로 인한 데이터 유실 복구 목적으로 한정(단, 코드를 기기 밖에 미리 저장해두지 않았다면 기기/저장소 자체를 잃어버리는 경우까지는 복구되지 않는다는 점도 함께 안내).
- **보관 및 관리**: 최초 설정 시 사용자가 직접 보존 기간(무기한/1년 미접속/6개월 미접속)을 선택하며, 설정 화면에서 언제든 변경하거나 즉시 수동 삭제할 수 있다는 점을 명시(섹션 B "보존 정책" 참조). 법령상 강제되는 기간이 아니라 사용자가 스스로 정하는 것임을 분명히 한다. `recovery_backups` 테이블에는 조회키(코드의 해시)와 암호문, 보존 기간 선택값, 갱신 시각만 저장되며 이메일·로그인 계정 등 사용자를 특정할 수 있는 다른 필드를 함께 수집하지 않음을 명시(웹서버 자체의 접속 로그 등 인프라 레벨 로깅 정책은 이번 스코프에서 다루지 않으므로, 이 항목의 "수집하지 않음"은 애플리케이션 데이터베이스에 한정된 서술로 좁혀 쓴다).
- 기존 피드백 데이터 관련 조항(수집 항목, 보관 후 삭제 등)은 변경 없이 유지.

> 참고(법적 조언 아님): 개인정보보호법 제21조에 따른 일반 파기 원칙(보유 목적 달성 시 지체 없이 파기)은 적용되나, 구 제39조의6(1년 미접속 시 강제 파기·분리보관)은 2023.9.15. 시행 개정으로 삭제되어 현재 적용되지 않는다. 실제 공개 서비스로 확장할 경우 변호사 검토 권장.

## 테스트 방법

- **스키마 교체 후 서버가 실제로 기동하는지** — `database.py`를 신규 스키마로 바꾸고 `lifespan()`에서 옛 함수 호출을 제거한 뒤, 컨테이너를 처음부터 새로 띄워서 `no such table` 등의 에러 없이 정상 기동하는지 확인 (이번 리뷰에서 드러난 핵심 회귀 지점)
- 백엔드: `/api/demo-seed`가 여전히 정상 동작하는지, 삭제된 라우트가 실제로 404를 반환하는지, 정적 서빙이 유지되는지, `CoopCoepMiddleware` 제거/유지 여부를 결정한 대로 kvvfs 저장이 실제로 정상 동작하는지 확인
- 복구코드: 코드 생성 → 푸시 → (다른 브라우저 프로필/시크릿 창으로) 같은 코드로 복원 → 데이터 일치 확인. 서버 DB의 `ciphertext` 컬럼을 직접 열어봤을 때 평문이 보이지 않는지 확인
- 오프라인 시나리오: 네트워크 차단 상태에서 CRUD가 정상 동작하고, 백업 푸시만 실패 후 재연결 시 다음 스로틀 주기에 재시도되는지 확인
- PWA에서 정상적으로 `/api/recovery/*`가 동작하는지 확인(같은 오리진이므로 이번 세션 검증 대상). Capacitor 네이티브 빌드는 절대 URL 설정 전까지 이 기능이 동작하지 않는 것이 "정상"임을 인지하고 별도로 실패 처리(예: 조용히 스킵)하는지 확인
- 보존 기간: `retention_months`를 짧게(예: 테스트용으로 분 단위 등 임시로 좁혀서) 설정한 뒤, 다음 `PUT` 요청 시 조건에 맞는 오래된 행이 실제로 삭제되는지, 무기한(NULL)으로 설정한 행은 절대 삭제되지 않는지 확인
- `DELETE /api/recovery/{lookup_key}` 호출 후 같은 코드로 `GET`이 404를 반환하는지 확인
